Test Suite
Introduction
The SCORM 2004 4th Edition (Ed.) Test Suite Version 1.1.1 [Referred to hereafter as the Test Suite] allows organizations to perform self testing on learning management systems (LMSs), Sharable Content Objects (SCOs) and content packages. The Test Suite performs testing to help organizations determine their conformance to SCORM 2004 4th Ed. Note: This version of the Test Suite is built to assess conformance with SCORM 2004 4th Ed. The use of this software DOES NOT IMPLY certification to SCORM 2004 4th Ed. For details on conformance and/or certification, refer ADLNet.gov.

SCORM defines a web-based learning "Content Aggregation Model" and "Run-Time Environment" for content objects. SCORM integrates a set of related technical standards, specifications, and guidelines designed to meet SCORM's high-level requirements-accessible, interoperable, durable, and reusable content and systems. SCORM content can be delivered to your learners via any SCORM-conformant LMS using the same version of SCORM.

SCORM specifically addresses a common:
  1. Specification for a Content Aggregation Model which includes content packaging.
  2. Method for representing the intended behavior of an authored learning experience such that any LMS will sequence discrete learning activities in a consistent way.
  3. Specification for a Run-Time Environment (RTE) that facilitates interoperable communication between LMSs and course content components.
The Test Suite assesses SCORM conformance by providing four testing components:
  1. LMS Conformance Test
  2. Content Package Conformance Test
  3. SCO RTE Conformance Utility Test
  4. Manifest Utility Test.

The Test Suite provides a common framework for all of the conformance tests. Each test includes a step-by-step set of detailed instructions as well as a log of each test action and the outcome.

The Test Suite runs within a Web browser on one local computer. All of the software is loaded on the machine and requires no HTTP Web server to run.

Installation Instructions

Read the entire Test Suite Readme prior to beginning the installation.

Prerequisites
This release of the Test Suite requires either the Java Development Kit (JDK) or the Java Runtime Environment (JRE) from either of the following Java versions:
  • Java SE 6, Update 1 or later
The latest version of this software is available for download free-of-charge at http://java.sun.com/javase/downloads/index.jsp. Archived versions are available at http://java.sun.com/products/archive/. Refer to this site for a list of specific Java versions verified.
Constraints
In addition to the preceding Java prerequisites, the Test Suite requires HTML and JavaScriptTM.

As of this release, the Test Suite was tested on Microsoft® Windows® Vista SP1 and XP SP3 on the Intel Pentium Platform using Microsoft Internet Explorer 7 and Internet Explorer 8.

The Test Suite has not been tested formally with any other browsers on any other platforms than those indicated above.

Installation

The Test Suite is available for download on the ADLNet.gov Web site as a self-extracting executable install file. After you download the file:

  1. Double-click the downloaded self-extracting executable file to begin the installation process.
  2. Click Next to continue from the "Welcome" page.
  3. Read the "License Agreement" and
    • Click Yes to agree with the terms of the license and continue with the install.
    • Click No if you do not agree with the terms of the license and want to exit the install.
  4. The recommended root installation directory is "C:\ADL". You may choose another root installation directory, but the Test Suite install will always append the "\SCORM_2004_4th_Ed_TS_V1.1.1" sub-directory to the specified root installation directory. The remainder of this Readme document assumes that the Test Suite will be installed to "C:\ADL\SCORM_2004_4th_Ed_TS_V1.1.1". This is referred to as "<install_directory>". If you override the default to use a different root installation directory, you must account for that when reading the remainder of this Readme.
  5. Click Next.
  6. The install will add program icons to the Program Folder of your Start menu. Type a new folder name, or select one from the existing list of Start Menu folders. "ADL" is the recommended folder. Click Next to install the Test Suite and the installation program will copy the necessary files to the appropriate directories.
  7. Click Finish to complete the installation.

The installation process adds a shortcut for running the Test Suite to your desktop. It also adds the following shortcuts to the "Programs" folder of your Start menu:

  • Link to ADLNet.gov
  • Shortcut to this Readme document
  • Shortcut to the Test Suite

The installation will not remove or un-install previous versions of the SCORM Test Suite. If you want to remove a previous version:

  1. Open the Windows "Control Panel".
  2. Open "Add or Remove Programs".
  3. Highlight the "SCORM Test Suite <version>".
  4. Click Change/Remove; and follow the instructions.
The LMS Test Content Packages are not included in the install file. These files are needed to run the LMS Conformance Test. You can download the LMS Test Content Packages separately at the ADLNet.gov Web site.

Configuration

To ensure that the Test Suite runs properly after it is installed:

  • The Internet browser must be set to check for a new page every visit to the page. To configure the browser:
    • For Microsoft Internet Explorer 6 Service Pack 2
      1. Open a Microsoft Internet Explorer browser.
      2. Click "Tools" then "Internet Options".
      3. On the "General" tab, within the "Temporary Internet Files" sub-section, click Settings.
      4. Ensure that the "Every visit to the page" radio button is selected.
      5. Click OK and then OK again.

    • For Microsoft Internet Explorer 7 and 8
      1. Open a Microsoft Internet Explorer browser.
      2. Click "Tools" then "Internet Options".
      3. On the "General" tab, within the "Browsing history" sub-section, click Settings.
      4. Ensure that the "Every time I visit the webpage" radio button is selected.
      5. Click OK and then OK again.

  • The Java Plug-in must be configured to disable caching. To disable the Java Plug-in cache:
    • For Java SE 6
      1. Open the Windows "Control Panel".
      2. Double-click the "Java" icon.
      3. Select the "General" tab.
      4. Under the "Temporary Internet Files" section, click Settings.
      5. Ensure the "Keep temporary files on my computer." checkbox is unchecked.

  • Ensure that the Sun Java Plug-in is selected as the default Java Environment for Internet Explorer. To set the default browser Java Environment:
    1. Open the "Java Plug-in" applet in the Control Panel.
    2. Select the "Browser" tab.
    3. Select the checkbox next to the browser that you are using to run the Test Suite.
    4. Click Apply.

  • If you are using Microsoft® Windows® XP SP3 these additional actions may be required:
    • Allow JavaScript to run by:
      1. Opening an Internet Explorer browser.
      2. Clicking "Tools" then "Internet Options".
      3. Check the box that states "Allow active content to run in files on My Computer" on the "Advanced" tab, under the "Security" section.
    • Allow the Test Suite to pop-up windows by either disabling pop-up blocking or excluding the Test Suite from pop-up blocking.

This completes the installation of the Test Suite software. Review the "Running the Conformance Tests" and "Known Issues" sections of this Readme prior to running the Test Suite. This Readme is installed to the installation folder specified in step #4 above. (e.g. "<install_directory>\SCORM_2004_4th_Ed_TS_V1.1.1\TestSuite")

Uninstalling the Test Suite

There are two methods that can be used for uninstalling the Test Suite:

  • Uninstall the Test Suite using the Windows Control Panel.
    • For Windows XP, the Windows Control Panel "Add/Remove Programs" feature can be used.
      1. Open the Windows "Control Panel"
      2. Double-click "Add or Remove Programs".
      3. Highlight "SCORM 2004 4th Edition Test Suite Version 1.1.1".
      4. Click Remove and follow the instructions
    • For Windows Vista, use the Windows Control Panel "Programs and Features" feature.
      1. Open the Windows "Control Panel"
      2. Double-click "Programs and Features".
      3. Highlight "SCORM 2004 4th Edition Test Suite Version 1.1.1".
      4. Click Uninstall on the toolbar above the list and follow the instructions
  • If the Test Suite is already installed, running the install again will uninstall the software. This method will not re-install the Test Suite. You will need to run the install a third time to accomplish a re-installation.

Note: Test Suite may not appear in the "Add/Remove Programs" menu. If it does not appear, use the second method to uninstall.

To run the Test Suite, open the < install_directory>\TestSuite\main.htm" file in Internet Explorer, or use the shortcuts added to the desktop during the installation. Select the test you want to perform from the main menu.

Click the "About the ADL SCORM 2004 4th Ed. Conformance Test Suite" link to view information about how the Conformance Test Suite works, and how to operate the Test Suite.

Each individual test offers step-by-step instructions, as well as an "About this Test" link that enables you to view detailed documentation about each test. The following section describes how to operate each test in detail.

Running the LMS Conformance Test
To run the LMS Conformance Test, click the corresponding link from the main Test Suite menu. Java Plug-in Security Warning dialog boxes will be displayed. This is because the Test Suite is reading and writing files from/to the local file system via Java Applets. Click Grant this Session or Grant Always to allow the Test Suite applet to run. Note that clicking Deny will cause the Test Suite software to not function properly.

If this is not the first time the LMS Conformance Test is run, prior to beginning the LMS Conformance Test ensure that the LMS Test Content Packages are fully removed from the LMS system and all global objectives associated with these content packages are removed.

Note: Check the Java Console after starting the LMS Conformance Test. If there are any Java exceptions reported, close all browser windows and start the test again.

LMS Overview
The LMS Conformance Test verifies that a test subject LMS is conformant with SCORM according to the SCORM 2004 4th Ed. Testing Requirements. This test will verify that the LMS is able to:
  • Import a known SCORM 2004 4th Ed.-Conformant Content Package

  • Launch an Asset

  • Launch SCORM 2004 4th Ed.-Conformant SCOs

  • Support the Run-Time Environment Application Program Interface (API) defined in SCORM

User Interface (UI) Tests
The SCORM 2004 4th Ed. Test Suite includes several new tests in the LMS Conformance Test for testing LMS provided UI Controls. These tests can be categorized as:

User Interface Inspection Tests
A number of LMS Test Content Packages were added or modified to include UI Inspection Tests. The content packages ask a series of questions about the LMS provided UI. When one of these packages is encountered, respond to the questions based on the current state of the LMS UI. Use the following terms to answer the questions:

  • Visible - A UI device exists and is either Enabled or Disabled:
    • Enabled - a UI device can be triggered by the learner to invoke a given navigation event. In context of the TOC, "Enabled" is analogous to "Selectable".
    • Disabled - a UI device cannot be triggered by the learner to invoke a given navigation event. A non-existent (Hidden) UI device is also considered "Disabled".
  • Hidden - No UI device exists. The learner shall not be able to invoke the navigation event corresponding to the hidden UI device.

User Interface Behavior Tests
A number of LMS Test Content Packages were added or modified to include a number of manually navigated LMS Test Content Packages. You will be prompted to sequence through the LMS Test Content Package manually using the LMS provided UI. When one of these packages is encountered, the LMS Test Content Package will indicate the expected navigation event by both popping-up an alert with the instructions and by writing the instructions in the test SCO. If at any time, it is not possible to trigger the specified Navigation Event from the LMS provided UI, then it will not be possible to continue with the LMS Conformance Test. At that time, abort the LMS Conformance Test by clicking Abort. The LMS is not SCORM conformant due to navigation limitations in its provided UI.

Content Package Overview

The Content Package Conformance Test verifies that the content package test subject complies with the SCORM 2004 4th Ed. Testing Requirements.

The purpose of this test is to verify that the Content Package test subject:

  • Contains an IMS Manifest at the root of the content package that:
    • Is named imsmanifest.xml.
    • Is well-formed.
    • Validates against the IMS Content Packaging XML Schema Definition (XSD) Version 1.1.4 ( imscp_v1p1.xsd found at http://www.imsglobal.org/ ).
    • Complies to the rules specified in the SCORM 2004 4th Ed. Content Package Application Profiles.
    • Uses extensions elements and if they are used, they are valid to their respective controlling document.
    • Verifies that the sequencing information is valid according to the IMS Simple Sequencing XML Schema Definition (XSD) ( imsss_v1p0.xsd and supporting files found at http://www.imsglobal.org/ ) if the IMS Manifest contains IMS Simple Sequencing Information.
    • If the IMS Manifest contains SCORM CAM Content Packaging extensions, verifies that the content packaging extensions are valid according to the SCORM 2004 4th Edition Content Packaging Extensions XML XSD Version 2.0 ( adlcp_v1p3.xsd found at http://www.adlnet.gov/ ).
    • Verifies that the sequencing extensions are valid according to the SCORM 2004 4th Ed. Sequencing Extensions XML XSD Version 2.0 ( adlseq_v1p3.xsd found at http://www.adlnet.gov/ ) if the IMS Manifest contains SCORM CAM Sequencing extensions.
    • Verifies that the navigation extensions are valid according to the SCORM 2004 4th Edition Navigation Extensions XML XSD Version 1.0 ( adlnav_v1p3.xsd found at http://www.adlnet.gov/ ) if the IMS Manifest contains SCORM CAM Navigation extensions.
  • Contains all supporting controlling documents required for XML Parsing at the root of the content package.
  • Contains optional Metadata XML Instances that are well-formed and valid according to the corresponding provided Controlling Document (e.g., XSD, DTD).
  • Contains conformant SCOs as determined by the SCO RTE Conformance Utility Test.

SCO Utility Overview
The SCO RTE Conformance Utility Test verifies that the test subject SCO is conformant with the SCORM 2004 4th Ed. Testing Requirements. This test will verify that the test subject:
  • Can be launched by a known conformant LMS

  • Supports the Run-Time Environment Application Program Interface (API) functions defined in SCORM 2004 4th Ed.

    Required
    • Initialize("")
    • Terminate("")
    Optional
    • Commit("")
    • GetValue("parameter")
    • SetValue("parameter_1","parameter_2")
    • GetLastError()
    • GetErrorString("parameter")
    • GetDiagnostics()
In addition to verification of the above requirements, this test will also audit which Run-Time Environment Data Model elements the SCO used in the content implementation, and verify that data elements are used in the correct manner with regard to the data type requirements associated with the data model elements.
Operational Instructions
  • The test can be run on a Single SCO or Multiple SCOs. You can also save or load a list of SCO file paths to and from a text (.txt) file.
    1. Provide the SCO Title, SCO Version/Release Number and SCO Vendor/Developer.
    2. Click Continue.
    3. Enter a timeout period (default is 20 seconds). This is the number of seconds the Test Suite will wait for the API method Initialize("") to be called. If the call is not made within this time, the Test Suite will end and the SCO will be given a conformance label.

      Note: If there is a possibility that the SCO may not call API method Initialize("") in the first 20 seconds, change this default value. No warning or error will be logged for changing this value.
    4. Click Browse to select the SCO that will be tested or a saved .txt file containing a list of SCO file paths.
    5. You may click Add to List to add another SCO or .txt file to the list.
    6. Click Continue after adding SCOs to the list.

      Note: If a file remains in the file input box, it will be appended to the list after you click Continue.

    7. Optionally, you may enter initialization values for any of the following elements: Learner ID, Learner Name, Credit/No-Credit, Mode, Comments From LMS, Launch Data, Objective Data, Scaled Passing Score, Max Time Allowed, Time Limit Action or Completion Threshold. These values are typically defaults that are provided by the LMS or via a Manifest.

      Note: If entering Comments From LMS or Objective Data, click the corresponding Add button, enter the values for the initial element of the array and click the corresponding Set button. After adding the last element of the list, click Done.

    8. Click Continue.
    9. Then click Launch SCO.
    10. Follow the necessary steps that are presented by the SCO to exercise the content.
    11. Click Complete Test in the Instructions frame to finish the test.
    12. If multiple SCOs were entered you will repeat steps 9 through 11 for each SCO to be tested.

  • After you click Launch SCO the Test Suite will launch the current SCO.

  • The Test Suite will then wait for a call to Initialize(""). The Test Suite will either wait for the call or the timeout period to expire (whichever comes first).

  • The Test Suite will wait for calls to any of the API functions listed below.

  • For Single SCO Testing: After the SCO RTE Conformance Utility Test surrogate LMS receives a call to Terminate(""), the SCO RTE Conformance Utility Test will end. Click Complete Test after this test ends to close the SCO window.

  • For Multiple SCO Testing: After the SCO RTE Conformance Utility Test surrogate LMS receives a call to Terminate(""), the current SCO RTE Conformance Utility Test will end. Click Complete Test to close the current SCO window. Processing will begin for the next SCO and will continue until the last SCO is tested.

  • Use Abort Test if a call to Terminate("") is not received from the SCO being tested. Wait for a period of time to see if the call will be invoked. If not, Abort Test will send a call to Terminate("") and end the testing, marking the SCO as being non-conformant.

  • At the end of a successful test the Test Suite will send to the Test Suite Log a set of supported features. These indicate what SCORM features a SCO supports. SCORM 2004 4th Ed. features are:

    API Support:
    • Find an LMS provided API Instance

    Session Methods:
    • Initialize()
    • Terminate()

    Data Transfer:
    • GetValue()
    • SetValue()
    • Commit()

    Support Methods:
    • GetLastError()
    • GetErrorString()
    • GetDiagnostic()

    Data Model Support:
    • SCORM 2004 4th Ed. Run-Time Environment Data Model

Checksum Utility Overview

The Checksum Utility Test verifies that the identified content package generated the summary log or the checksum value from the summary log during a successful run of the Content Package Test.

Operational Instructions
  1. Click Browse... and select the content package whose log or checksum value authenticity you want to evaluate. Select only the zip (PIF) format of the content package or the imsmanifest.xml file from an extracted (non-PIF) format of the content package.
  2. Select the method of providing the checksum value, either manually or by choosing the summary log.
  3. If you chose the Checksum Value option, enter the checksum value in the text box as it appeared in the Content Package Test Summary Log. If you chose the Log File option, click Browse... to select the XML summary log file created by the Content Package Test for the given content package.
  4. Click Compare.
  5. The result of the Checksum Utility Test will appear in the box below the Compare button. A message stating "Checksum Successful" indicates that the given checksum value corresponds to the given SCORM content package. A message stating "Checksum Failed" indicates that the given checksum value does not correspond the given content package. This means you cannot assume the package adheres to the SCORM Testing Requirements.
Operational Notes
  • The Checksum Utility Test can be used only with Content Package Test logs created with the SCORM 2004 4th Edition Test Suite Version 1.1 and SCORM 2004 4th Edition Test Suite Version 1.1.1
  • The Checksum Utility Test can be run multiple times with different SCORM content packages and/or checksum values without closing the browser window.
  • The checksum value is generated by applying a hashing algorithm against all the files within the SCORM content package. Any change to the SCORM content package, such as adding, removing, or modifying files will invalidate the checksum value generated from a previous Content Package Test run.
  • If the SCORM content package needs to be modified, for example to remove files marked as excess baggage or to update resource locations within the manifest, retest the package using the Content Package Test to obtain a new checksum value.

Known Issues
  • For security reasons Microsoft Internet Explorer requires that content that performs cross frame or cross-window scripting be loaded from the same domain ( http://msdn2.microsoft.com/en-us/library/ms533028.aspx). Because of this restriction, when testing a SCO in the SCO RTE Conformance Utility Test, the SCO that is being tested must be from the same domain as the domain for which the Test Suite is running. The SCOs use JavaScript to locate the LMS-provided API Instance, which requires the use of cross frame and/or cross-window scripting. For example, if the Test Suite is installed on your local machine on the C: drive, the SCO must be physically located on the C: drive also.

  • The Content Package Conformance Test is unable to test a content package using the http or ftp protocol.

  • When starting the LMS Conformance Test, check the Java Console. If there are any Java exceptions reported, close all browser windows and start the test again.

  • The links to the Detailed Logs contained in the Summary Logs will not be fully functional until the Test Suite completes the entire test.

  • Due to the design of the IMS Content Packaging Version 1.1.4 XML Schema Definition (XSD), an Internet connection may be required to perform the schema validation test in the Content Package Conformance Test.

  • Close the browser completely after each test before starting another Test Suite test.

  • If LMS Conformance Test is closed and reopened while having another internet browser open, the JVM may not be shut down and the socket may not be released. If this happens, the Applet that utilizes the socket throws an exception and fails to start. The following exception will be found in the Java Console if it encounters this issue:
    • SEVERE: Exception caught: java.net.BindException: Address already in use: JVM_Bind
      java.net.BindException: Address already in use: JVM_Bind
    Close all internet browsers, ensure that no JVMs are running (no Java icons are found in the task manager), and restart the Test Suite.

  • It is possible for a java.io.IOException to be thrown during the second activity of LMS Test Content Package DMI. The exception stems from issues with certain versions of the Java JRE and the joda.time plugin. The exception does not have any affect on conformance or the operation of the Test Suite.

  • When running the Test Suite Content Package Test or Manifest Test, the package will fail if the manifest or a schema listed in the manifest directly references a schema or sub-schema in a directory with non-English characters. For the purpose of this issue, a non-English character is any character outside of the Basic Latin Unicode code chart.

      • Filename
        • The path of a PIF(.zip) Content Package, the path of a non-PIF(.xml) Content Package manifest file, or the path of a directory containing PIF(.zip) Content Package files.
        • Multiple filenames can be provided to allow sequential validation of multiple Content Packages or Content Package manifests.
      • Application Profile Type
        • The given file(s) or folder contents can be validated against the Resource Application Profile if the option -r is entered after the file(s) or folder(s).
        • If no Application Profile Type option is provided, the Manifest Tester Utility, by default, will validate the entered file(s) and or folder contents against the Content Aggregation Application Profile.
      • Help
        • A brief summary of the Manifest Tester Utility instructions will be displayed at the command prompt if only the -h or -help options are entered.
  • Examples
    • java -jar adlvalidator.jar C:\MyFiles\MyContentPackage.zip
    • java -jar adlvalidator.jar C:\MyFiles\MyContentPackage1.zip C:\MyFiles\MyContentPackage2.zip
    • java -jar adlvalidator.jar C:\MyFiles\MyResourceContentPackage.zip -r
    • java -jar adlvalidator.jar C:\MyFiles\MyContentPackage.zip C:\MyFiles\nonPIF\imsmanifest.xml
    • java -jar adlvalidator.jar "C:\MyFiles\My ContentPackages\\"
